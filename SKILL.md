# 下厨房食谱提取器

使用有头且持久化的 Playwright 浏览器打开下厨房食谱页面，完成必要的人机验证，读取页面内容，并输出便于阅读和复用的食谱摘要。

## 地址输入规则

直接调用本 skill 时，先判断当前消息是否包含完整的食谱 URL：

- 已包含完整 URL：提取 URL，确认域名和路径后直接进入浏览器流程。
- 未包含 URL：只发送下面的文本提示，然后等待用户下一轮回复；不启动浏览器、不猜测 URL、不提取内容：

  > 请把完整的下厨房食谱地址发给我，例如 `https://www.xiachufang.com/recipe/123456/`。收到后我会整理食材、做法和小贴士。

- 只收到关键词、短链接、分享文案或不完整地址：请用户补充“完整网页地址”，不要猜测食谱页面。
- 收到地址后，去掉末尾多余标点并确认域名和路径；确认有效后再继续执行提取。
- 仅接受 `https://www.xiachufang.com/recipe/...` 或下厨房子域名下明确的食谱页面。若用户给的是其他网站地址，说明当前 skill 只支持下厨房，并请其提供下厨房链接。

## 工作流程

1. 获取并确认地址输入阶段得到的食谱 URL。没有确认到完整 URL 前，不要启动浏览器。
2. 启动浏览器并打开页面：
   ```bash
   playwright-cli open --headed --persistent "<食谱URL>"
   ```
   如果当前 Playwright 会话已经打开，则使用 `playwright-cli goto "<食谱URL>"`，但首次打开任务仍应优先使用 `--headed --persistent`。
3. 用 `playwright-cli snapshot` 检查页面状态。
4. 如果出现“请滑动完成验证”等人机验证：
   - 使用 `playwright-cli screenshot --filename=/tmp/xiachufang-captcha.png` 查看验证控件位置。
   - 通过有头浏览器的正常鼠标操作完成滑块拖动，例如 `mousemove`、`mousedown`、多次 `mousemove`、`mouseup`。
   - 根据当前截图定位滑块，不要硬编码某个页面的固定坐标。
   - 不要尝试绕过、破解或伪造验证；如果验证无法完成，向用户说明需要其在已打开窗口中手动完成。
5. 验证通过后再次运行 `playwright-cli snapshot`，读取以下页面信息：
   - 菜谱标题、作者、评分、制作人数及发布日期（页面有则提取）。
   - “用料”表格中的食材与用量。
   - “做法”部分的全部步骤及关键时间、火候。
   - “小贴士”中的注意事项、替代方案和解冻/处理建议。
6. 如果正文没有完整出现在快照中，先滚动页面再快照；必要时使用 `playwright-cli eval` 读取页面可见文本或表格内容。不要只依赖页面标题或搜索结果。
7. 将结果整理后直接回复用户。除非用户另有要求，不要自动写入本地文件。
8. 确认食谱信息已经完整读取并整理后，回复用户前执行：
   ```bash
   playwright-cli close
   ```
   页面关闭是固定收尾步骤，不要在成功提取后继续保留浏览器窗口。

## 输出格式

默认使用以下结构，缺失字段可省略：

```markdown
## 菜谱名称

来源：[下厨房原菜谱](URL)
作者：
评分/制作人数：

### 食材
- 食材：用量

### 做法
1. ...
2. ...

### 小贴士
- ...
```

按页面事实进行准确转述；食材名称、数量、时间和火候不要擅自推测。步骤可在不改变含义的前提下压缩重复描述，保留关键操作顺序。若原页面明确说明盐、油或其他调味料未列出，也要在小贴士中注明。

## 异常处理

- 页面打不开或网络失败：报告具体 URL 和失败状态，建议用户重试，不要编造食谱内容。
- 触发登录但页面正文可见：无需登录，继续读取公开内容。
- 触发验证码且自动操作失败：保留有头浏览器窗口，请用户完成验证后再继续。
- 页面内容疑似不完整：明确标注“页面未显示完整”，只输出已确认内容。
- 如果页面已成功打开但在提取过程中发生错误，结束本次尝试前也应执行 `playwright-cli close`；只有等待用户手动完成验证码时暂不关闭页面。
- 页面版权声明：以整理和概述为主，不大段逐字复制页面原文；需要引用时保持简短并附原页面链接。
